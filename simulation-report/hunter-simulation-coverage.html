2 · Locks, Deadlocks & Blocking Queries

15 total  •  3 achieved  •  12 not achieved

Why the count looks low: run_all.sh fires all 15 drills concurrently against shared tables by design. In the one full-scale run with evidence (2026‑07‑09, 20:26 UTC), script 04's table-wide LOCK TABLE landed first and became the root of a 22‑session blocking chain that absorbed most of the other drills' sessions as blocked bystanders — before their own distinct failure signature (a deadlock cycle, a genuine idle-in-txn state, a row-level lock) could register. This is a run-design artifact, not evidence that those checks are undetectable in isolation. The most recent fast-scale run (default ~6‑8s holds, well under the hunter's 300s poll interval) shows 0 issues, independently confirming the README's own documented reliability caveat.

Methodology

"Not achieved" covers four distinct situations, deliberately kept separate above rather than collapsed into one verdict: a genuine detection-pipeline gap, an environment/data limitation in the dev instance, a check that is owned by a different hunter than the one the folder name suggests, and one confirmed script bug.
